# Architecture & engineering notes

A deeper look at how the substation design generator is built, for readers who want
more than the README. The source is private; this is a description, not a code dump.

## Pipeline overview

```
Plot + parameters
      │
      ▼
Geometric layout engine  ──►  Generation result (WGS84 geometry, equipment, zones)
      │                              │
      │                              ├─►  Electrical layer      (Z-bus solver)
      │                              ├─►  Protection settings   (ANSI 50/51/87T/49)
      │                              ├─►  Cable-channel router  (comb + orthogonal A*)
      │                              ├─►  SCL export            (IEC 61850-6 SSD/SCD)
      │                              └─►  Norms copilot context  (LLM, no math)
      ▼
Persisted project (MariaDB) ──► REST ──► Vue 3D dashboard
```

## 1. Geometric layout engine

The generator is fundamentally a constraint-respecting geometric search. It
places primary equipment (transformers, HV breaker bays, section switchgear,
metal-clad switchgear, control building, busbars) on the plot so the result honours:

- electrical clearances and equipment security zones,
- fire gaps and road access,
- the fence setback required by PUE / SP.

Candidate core layouts are searched and scored; the winner is assembled with full
detail (conductors, cable channels, control building). All math is done in a
local coordinates using the JTS Topology Suite, then projected back to
WGS84 for rendering and storage.

### Topological invariants

The domain enforces a fixed topology:

The algorithm implements a sequential process for generating the equipment layout of a standard step-down substation, such as a 110/10 kV substation.
- The generation process begins by determining the positions of two power transformers using a genetic algorithm. 
Candidate layouts are evaluated based on the relative positions of the transformers, the required clearances, and sufficient space for the equipment to be placed during the subsequent stages. 
The transformer types are selected from a predefined catalogue of existing transformer models with their actual dimensions, including oil containment pits.
- Once the transformers have been positioned, the algorithm generates the transformer-side circuit breakers and their disconnectors. 
It then places the high-voltage circuit breakers and disconnectors associated with the incoming power lines, attempting to orient them towards the substation boundaries.
- Next, a bus-switch circuit breaker and its disconnectors are generated between the two busbar sections. Two voltage transformers are also placed, one on each high-voltage busbar section.
- The Outdoor Metal-Clad Switchgear(KRUN) is then generated on the side opposite the transformers, taking all required clearances into account. 
The total number of feeder panels, including the presence and number of reserve panels, is calculated from the parameters specified by the user.
- The control building is generated next, followed by the cable trench system, whose layout is designed to resemble the arrangement typically used at real substations.
- When a user requests multiple layout variants, they are generated by positioning the transformers relative to different substation boundaries, starting with the longest boundary.


## 2. Electrical layer — short-circuit analysis

The generated (geometric) model has no electrical model on its own, so a separate
layer builds one: a network of nodes and per-unit impedance branches
(source system → HV → transformers → LV), assembled per switching regime.

- The network is solved via the bus impedance matrix - Thévenin equivalent of a complex multi-node network, `Z-bus = Y-bus⁻¹`, using a
  small hand-written complex-matrix layer.
- A fault at bus *k* gives `I = c / Z_kk`, bus voltages `V_i = c·(1 − Z_ik/Z_kk)`, and
  branch contributions from the currents on adjacent branches.
- Switching regimes A/B/C/D (section breakers open/closed on the HV and LV sides)
  are each solved; closed section breakers merge nodes via union-find.
- Two standards share a `ShortCircuitStandard` abstraction:
  - **GOST R 52735-2007** — base voltage = average voltage, c = 1.0.
  - **IEC 60909** — base voltage = nominal, c_max = 1.10, transformer impedance
    correction factor `K_T = 0.95·c_max / (1 + 0.6·u_k/100)`.
  The two reconcile to the expected ratios (HV: c·U_avg/U_nom exactly; MV a bit higher
  due to the base change + Kt).
- Transformer reactance is derived properly from `u_k%` and `ΔP_k`
  (`|Z_T| = u_k/100 · S_base/S_n`, `R_T = ΔP_k·S_base/(1000·S_n²)`,
  `X_T = √(|Z_T|² − R_T²)`), so the source is not treated as purely reactive and the
  impulse factor κ on the HV side is not over-estimated.

Outputs per fault: `I⁽³⁾`, `I⁽²⁾ = (√3/2)·I⁽³⁾`, impulse current, κ, X/R, per-unit
R/X, bus voltages and branch contributions.

## 3. Protection settings

From the fault currents the tool proposes preliminary settings for the transformer
protections and checks selectivity:

- **51 (IDMT), 50 (instantaneous), 87T (differential), 49 (thermal)** + LV incomer.
- **De-rating uses the maximum regime** (all sources in, c_max, fault through one
  transformer); sensitivity for relay protection uses the minimum regime (c_min, two-phase fault) —
  the two extremes are computed from real regimes, not a 0.866 multiplier.
- Sensitivity Kч is referred to the HV side (`I_HV = I_MV·U_MV/U_HV`).
- **CT selection** is automatic: nearest standard IEC primary ratio ≥ 1.05·I_nom,
  5 A secondary, with a primary/secondary display toggle for the relay engineer.
- Every setting carries the formula with numbers substituted and a status
  (preliminary vs. simplified). The panel states clearly these are
  **preliminary values, not commissioning settings**; earth-fault and feeder
  protections are intentionally out of scope (they need zero-sequence, so it's not implemented right now).

## 4. Cable-channel routing

Secondary cabling is routed in two regimes:

- Inside each equipment security zone: a comb in the section's local frame —
  a longitudinal trunk along the outer edge of the equipment row and orthogonal taps
  from each apparatus.
- Outside the zone: an orthogonal A\* router on a 1 m grid aligned to the
  switchgear, whose obstacle map is the union of equipment buffers with a
  morphological closing to fill gaps, so a channel never squeezes between two
  apparatuses. It routes around the primary equipment to reach the switchgear and
  the control building.

## 5. IEC 61850 SCL export

The design exports to **IEC 61850-6** (edition 2.1 / schema 2007B4) as both an SSD
(system specification) and a SCD (with IEDs and communication):

- Primary equipment → conducting equipment: breaker→CBR, disconnector→DIS,
  CT→CTR, VT→VTR, arrester→SAR, line→IFL, transformer→PTR/PTW.
- ANSI functions → logical nodes: 51→PTOC, 50→PIOC, 87→PDIF, 49→PTTR, 79→RREC,
  86→PTRC; apparatus XCBR/XSWI/CSWI/CILO; per-phase TCTR/TVTR.
- Per bay: one controller IED (control + protection logical devices) and one
  merging unit IED (measurement + sampled values), on separate station-bus and
  process-bus subnetworks, with GOOSE interlocking and SMV subscriptions.
- A populated DataTypeTemplates (real DOs/DOTypes/DATypes/EnumTypes per CDC) and
  actual relay settings written into the protective LNs' DAIs.

The SCL model is a **hand-written, focused JAXB model** (not XJC-generated): small,
readable classes with the SCL namespace declared
at package level. Every export is validated against the official SCL XSD (from
`rte-france/SCL_Loader`) in the test suite.

## 6. Norms copilot

A floating LLM assistant explains design decisions and the derivation of each value.

- The LLM never calculates. The deterministic engine emits ready
  derivation strings (each setting's formula, each fault's derivation); the model
  cites them and points at the norm. Numbers never originate from the model.
- The norms corpus is small (a few KB), so it is provided as full context rather
  than retrieved via a vector store. Cards come from JSON configs, code enums (with
  PUE quotes) and a symbolic formula reference.
- Providers are routed (DeepSeek / GigaChat) behind an abstraction; any failure
  degrades to a warning, keeping the rest of the app usable.

## 7. Frontend

- **Vue 2.7** with the Composition API and `<script setup>`.
- **MapLibre GL 3.6** renders 3D equipment as `fill-extrusion` prisms; **deck.gl**
  (via `MapboxOverlay`) draws the phase conductors.
- Ground shadows are projected fakes — MapLibre's `fill-extrusion` doesn't cast
  shadows, so each footprint's shadow is a convex hull of its contour and a
  height-offset copy, aligned with the configured light direction, drawn under the
  equipment.
- A custom lightweight RU/EN i18n reactive.

## Rendering realism note

The 3D equipment is built entirely from extruded prisms — no glTF meshes — with
hand-built detail (ribbed porcelain support insulators, transformer expansion tank /
rails / bushings, control-building roof slab). glTF meshes (deck.gl
ScenegraphLayer) and three.js remain options for photorealism; the extrusion path was
chosen deliberately for control and performance.
